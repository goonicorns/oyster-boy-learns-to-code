Unix Shell

The shell is how you talk to your computer in text. You type commands, press Enter, and the computer does exactly what you asked — including things you didn't mean. That's what makes it powerful and occasionally dangerous. Take it slow at first.

⚠ DANGER ZONE — Shell Commands That Can Wreck Your Day

The shell does not have an undo button. There is no trash can. When you delete a file from the command line, it is gone. Not in Trash, not in a recycle bin — gone. Read this section carefully and refer back to it.

rm — Remove Files (The Wrecking Ball)

# Removes one file — reasonable, but still no undo
rm file.txt

# -r = recursive (go into all subfolders)
# -f = force (don't ask me, just do it, ignore errors)
# Together: wipes out a folder and everything inside it with zero confirmation
rm -rf foldername/

# CATASTROPHIC — deletes your entire home directory
# ~ means /Users/yourname — all your files, projects, photos, everything
rm -rf ~/

# SYSTEM KILLER — deletes the entire operating system
# The computer will not boot after this
rm -rf /

# Also very bad — "." means "right here"
# This wipes out your current folder and everything in it
rm -rf .

Before running any rm command: run ls first to see what's there. Triple-check the path. Never use -rf unless you are 100% certain. If in doubt, move the folder somewhere safe first with mv.

sudo — Running as Administrator

# sudo = "Super User DO" — runs the command with full system privileges
# Normal rm -rf can't hurt system files. sudo rm -rf CAN.
sudo rm -rf /etc          # Destroys system configuration
sudo chmod -R 777 /       # Removes all security from entire system

# Rule: never run a command with sudo that you don't fully understand.
# Never paste sudo commands from the internet without reading every word.

The Fork Bomb (know what it looks like so you never run it)

:(){ :|:& };:   # DO NOT RUN — spawns infinite processes, crashes the machine

If you ever see this in instructions from someone, close the terminal immediately. It's purely malicious. Even if told it "sets up your environment," it doesn't — it crashes your computer.

Overwriting Files with Redirects

# ">" overwrites — if the file exists, the old content is gone
echo "new content" > important-file.txt   # WIPES important-file.txt first

# ">>" appends — safe to add without destroying existing content
echo "new line" >> important-file.txt     # Adds to the end, nothing lost

Permissions

Every file has an owner and rules for who can read it, write to it, or execute it as a program.

# "ls -la" shows permissions as the first column:
-rw-r--r--  1 alice  staff   1234  Jun 12  notes.txt
drwxr-xr-x  3 alice  staff    128  Jun 12  projects/

# Breaking down "-rw-r--r--":
# First character:  "-" = file, "d" = directory, "l" = link
# Next 3:  rw-  = what the OWNER can do (read + write, not execute)
# Next 3:  r--  = what the GROUP can do (read only)
# Last 3:  r--  = what EVERYONE ELSE can do (read only)
# r = read, w = write, x = execute
CommandWhat it does
chmod +x script.shMake a file executable (so you can run it as a program)
chmod -x script.shRemove execute permission
chmod 644 file.txtOwner: read+write. Everyone else: read-only. Common for files.
chmod 755 folder/Owner: full control. Others: read+enter. Common for folders.
chmod 600 private.keyOnly the owner can read/write. Nobody else can even see it.
chown alice file.txtChange who owns a file (may need sudo)
ls -laSee all permissions in the current folder

Tmux

Tmux is a terminal multiplexer — it lets you split one terminal window into many panes, have multiple tabs (windows), and keep everything running even when you disconnect and walk away. It's like having a persistent workspace that lives on the server.

The Prefix Key: Almost every tmux command starts with pressing Ctrl+B, then releasing it, then pressing another key. This is called the prefix. So "Ctrl+B then d" means: hold Ctrl and press B, let go, then press D.

Golang — Full Walkthrough

Go (also called Golang) was made by Google. It's designed to be simple and hard to screw up — if something compiles (passes Go's checker without errors), it almost always does what you expect. No hidden magic. No guessing. The compiler yells at you for unused variables, missing error checks, and weird imports — all things that would cause subtle bugs in other languages.

How Code Becomes a Program

Before we write a single line, you need to understand what actually happens when you "run" Go code. This part confuses beginners because it's invisible — but it's happening every time.

The problem: Your computer's CPU doesn't understand Go. It only understands raw machine instructions — a very low-level language with no concepts like "string" or "function." So how do you get from code you can read to something the CPU can execute?

The compiler: The Go compiler (the go program you install) reads your Go code and translates it into machine instructions your CPU understands. This process is called compiling. The output is a binary file — a program your computer can actually run.

# Your Go source code (readable by you)
# ↓
# The Go compiler reads it and checks for errors
# ↓
# If there are errors, it tells you and stops
# If there are no errors, it produces a binary file
# ↓
# The binary file runs on your CPU (not readable by humans)

Why the compiler yells at you: Before your code can run, the compiler checks it for problems. If you use a variable you never declared, reference a function that doesn't exist, or try to add a number and a string — the compiler refuses to continue. This is actually good. It catches mistakes before they become runtime bugs (crashes or wrong output when actual users are running the program).

The two main ways to run Go:

CommandWhat it doesWhen to use
go run main.go Compiles and immediately runs — no file saved to disk Development and testing. Fast feedback loop.
go build -o myprogram Compiles and saves the result as a standalone executable file called "myprogram" When you want to distribute or deploy the program

Modules: Go organizes projects into "modules." A module has a go.mod file that names the project and tracks what external packages it uses. Every Go project starts with go mod init projectname. Think of it as the project's ID card.

# Start a new Go project
mkdir myproject && cd myproject
go mod init myproject    ↑ creates go.mod — run this once per project

# Check if Go is installed
go version               should print something like: go version go1.22 darwin/arm64

# Run a file directly
go run main.go

# Compile to a binary
go build -o myprogram    creates ./myprogram
./myprogram              run the binary

# Add an external package (downloads it and adds to go.mod)
go get github.com/some/package

# Format your code (always do this — Go has strict formatting rules)
go fmt ./...

# Run tests
go test ./...

# Check for common mistakes
go vet ./...
Install Go: Run go version to check if it's already installed. If you get "command not found", download it from golang.org and follow the install instructions for your OS. After installing, close and reopen your terminal.

Hello World — What Every Go File Looks Like

package main    // Every Go file starts with a package declaration.
              // "main" is special — it's what makes this a runnable program.

import "fmt"     // "fmt" = format. Gives us Println, Printf, Errorf, etc.
              // Import only what you use — Go will refuse to compile if you import unused packages.

func main() {                      // main() is where your program starts
    fmt.Println("Hello, World!")    // Print text followed by a newline
}
# Run without compiling (great for development)
go run main.go

# Compile to a binary you can run anywhere
go build -o myprogram main.go
./myprogram

# Format your code automatically (always run this — Go is opinionated about formatting)
go fmt ./...

Variables

A variable is a named container for a value. Give it a name, put something in it, use it later.

package main

import "fmt"

func main() {
    // Long form — explicit type declaration
    var name  string  = "Alice"
    var age   int     = 30
    var score float64 = 98.5

    // Short form — most common inside functions
    // Go figures out the type from what you assign (:= is "declare and assign")
    city   := "Tokyo"
    active := true

    // Declare multiple at once
    x, y := 10, 20

    // Constants — set once, never changed
    const MaxRetries = 3
    const Pi         = 3.14159
    const AppName    = "MyCoolApp"

    fmt.Println(name, age, score, city, active, x, y)
}
Important: Go refuses to compile if you declare a variable and never use it. This is intentional — it forces you to keep your code clean.

Types

Go cares about types — you can't mix an int and a string by accident. You have to explicitly convert.

// ── Integers (whole numbers) ─────────────────────────────────
var i   int   = 42       // Most common. Size depends on your OS (64-bit on 64-bit machines).
var i32 int32 = 42       // Always exactly 32 bits (-2B to +2B)
var i64 int64 = 42       // Always 64 bits (very large range)
var u   uint  = 42       // Unsigned: no negatives, but twice the positive range

// ── Decimals ─────────────────────────────────────────────────
var f32 float32 = 3.14   // Less precision (good enough for most things)
var f64 float64 = 3.14   // More precision. Default when you write a decimal.

// ── Text ──────────────────────────────────────────────────────
var s string = "hello"    // Text. Always double quotes. Can hold any Unicode.
var r rune   = 'A'       // Single character. Single quotes. (It's actually an int32 underneath.)
var b byte   = 65        // A single byte (0–255). byte = uint8.

// ── Boolean ───────────────────────────────────────────────────
var ok bool = true       // Only true or false. Never 1/0 or "yes"/"no".

// ── Zero values — what you get when you don't set a value ─────
var n   int     // 0
var f   float64 // 0.0
var str string  // "" (empty string, not null)
var bl  bool    // false

// ── Converting between types (Go won't do it automatically) ───
i   = int(f64)             // float64 → int (chops off decimals, no rounding)
f64 = float64(i)           // int → float64
s   = fmt.Sprintf("%d", i) // int → string ("42" not the number 42)
i, _ = strconv.Atoi("42") // string → int (returns an error too, hence the _)

Control Flow

if / else if / else

if age >= 18 {
    fmt.Println("You can vote")
} else if age >= 16 {
    fmt.Println("Almost there")
} else {
    fmt.Println("Too young")
}

// You can initialize a variable right in the if — it only exists inside the block
if err := doSomething(); err != nil {
    fmt.Println("error:", err)
}

for — the only loop in Go (but it does everything)

// Classic counter loop
for i := 0; i < 5; i++ {
    fmt.Println(i)  // 0 1 2 3 4
}

// "While" style — just a condition, no counter
n := 1
for n < 100 {
    n *= 2
}

// Infinite loop — runs forever until you break out
for {
    line := readLine()
    if line == "exit" {
        break  // exit the loop entirely
    }
    if line == "" {
        continue  // skip this iteration, go to next
    }
    fmt.Println(line)
}

// Loop over a slice — gives you index AND value
fruits := []string{"apple", "banana", "cherry"}
for i, fruit := range fruits {
    fmt.Printf("%d: %s\n", i, fruit)
}

// If you only want the value, use _ to discard the index
for _, fruit := range fruits {
    fmt.Println(fruit)
}

// Loop over a map (order is random every time — that's intentional in Go)
ages := map[string]int{"Alice": 30, "Bob": 25}
for name, age := range ages {
    fmt.Printf("%s is %d\n", name, age)
}

switch

switch day {
case "Monday":
    fmt.Println("Ugh")
case "Friday":
    fmt.Println("Almost there!")
case "Saturday", "Sunday":  // multiple values in one case
    fmt.Println("Weekend!")
default:
    fmt.Println("Somewhere in the middle")
}

// No variable — acts like an if/else chain
switch {
case score >= 90:
    fmt.Println("A")
case score >= 80:
    fmt.Println("B")
default:
    fmt.Println("Keep studying")
}

Functions

// Basic function — takes a string, returns a string
func greet(name string) string {
    return "Hello, " + name + "!"
}

// Multiple return values — very common in Go
// Usually: (result, error)
func divide(a, b float64) (float64, error) {
    if b == 0 {
        return 0, fmt.Errorf("cannot divide by zero")
    }
    return a / b, nil   // nil = "no error"
}

// Using a function with multiple returns
result, err := divide(10, 2)
if err != nil {
    fmt.Println("oops:", err)
    return
}
fmt.Println(result)  // 5

// Variadic — accepts any number of arguments
func sum(nums ...int) int {
    total := 0
    for _, n := range nums {
        total += n
    }
    return total
}
fmt.Println(sum(1, 2, 3, 4, 5))  // 15

// defer — runs at the very end of the function, no matter what
// Great for cleanup: closing files, releasing resources
func readFile(path string) error {
    f, err := os.Open(path)
    if err != nil {
        return err
    }
    defer f.Close()    // Will run when readFile() exits, even if it panics

    // ... read the file ...
    return nil
}

Slices — Ordered Lists

A slice is a list that can grow. It's the most common collection type in Go.

// Create a slice with initial values
names   := []string{"Alice", "Bob", "Charlie"}
numbers := []int{1, 2, 3, 4, 5}

// Create an empty slice and add to it
var scores []int                          // nil slice (length 0)
scores = append(scores, 95)              // [95]
scores = append(scores, 87, 92)          // [95, 87, 92]
scores = append(scores, numbers...)       // spread another slice in

// Make a slice with a specific size (all zeros)
zeros := make([]int, 5)    // [0, 0, 0, 0, 0]
buf   := make([]byte, 1024) // common for buffers

// Access by index — starts at 0, NOT 1
first := names[0]               // "Alice"
last  := names[len(names)-1]   // "Charlie"
names[1] = "Bobby"              // Modify in place

// Slice of a slice (shares underlying memory)
sub  := numbers[1:4]  // [2, 3, 4]  — index 1 up to (not including) 4
head := numbers[:3]   // [1, 2, 3]  — from beginning to index 3
tail := numbers[2:]   // [3, 4, 5]  — from index 2 to end
copy := numbers[:]     // full copy-reference of the slice

// Length and capacity
fmt.Println(len(names))  // 3 — how many elements are in it
fmt.Println(cap(names))  // capacity — how much space is allocated

Maps — Key → Value Lookup

A map is a dictionary. You look things up by key instead of by position.

// Create a map with initial values
ages := map[string]int{
    "Alice": 30,
    "Bob":   25,
}

// Create an empty map (you MUST use make, not {} for empty maps you'll add to)
config := make(map[string]string)

// Add or update
ages["Charlie"] = 35
config["host"] = "localhost"

// Read a value
aliceAge := ages["Alice"]   // 30

// IMPORTANT: check if a key exists before using it
// If you read a missing key you just get the zero value — no error
age, exists := ages["Dave"]
if !exists {
    fmt.Println("Dave is not in the map")
}

// Delete a key
delete(ages, "Bob")

// Loop over a map
for name, age := range ages {
    fmt.Printf("%s is %d years old\n", name, age)
}

// Pattern: count occurrences
words := []string{"a", "b", "a", "c", "b", "a"}
count := make(map[string]int)
for _, w := range words {
    count[w]++
}
// count is now: {"a": 3, "b": 2, "c": 1}

Structs — Custom Data Types

A struct groups related fields into one thing. It's Go's answer to "objects" — without the complicated class inheritance stuff.

// Define a struct type (usually at the top of the file, outside functions)
type User struct {
    Name  string
    Email string
    Age   int
    Admin bool
}

// Create one
alice := User{
    Name:  "Alice",
    Email: "alice@example.com",
    Age:   30,
    Admin: true,
}

// Access fields with a dot
fmt.Println(alice.Name)   // "Alice"
alice.Age = 31            // Modify a field

// Value method — gets a copy of the struct, reads it
func (u User) Greet() string {
    return "Hi, I'm " + u.Name
}

// Pointer method — works on the REAL struct, can modify it
func (u *User) Birthday() {
    u.Age++   // This actually changes alice.Age
}

fmt.Println(alice.Greet())
alice.Birthday()
fmt.Println(alice.Age)    // 32

// Pointer to a struct — common when passing structs around
alicePtr := &alice         // & means "give me a pointer to alice"
alicePtr.Age = 33          // Go auto-dereferences — no need for (*alicePtr).Age

// Create directly as a pointer
bob := &User{Name: "Bob", Age: 25}

// Nested structs
type Address struct {
    Street  string
    City    string
    Country string
}

type Person struct {
    Name    string
    Address Address
}

p := Person{Name: "Charlie", Address: Address{City: "Tokyo"}}
fmt.Println(p.Address.City)   // "Tokyo"

Interfaces — Describing Behavior

An interface says "any type that has these methods can be used here." It's how Go handles flexibility. You don't declare that you're implementing an interface — Go figures it out automatically based on your methods.

// Define an interface
type Speaker interface {
    Speak() string
}

// Any type with a Speak() method automatically satisfies Speaker
type Dog struct{ Name string }
type Cat struct{ Name string }
type Human struct{ Name string }

func (d Dog)   Speak() string { return "Woof!" }
func (c Cat)   Speak() string { return "Meow!" }
func (h Human) Speak() string { return "Hello!" }

// A function that accepts any Speaker
func makeSpeak(s Speaker) {
    fmt.Println(s.Speak())
}

makeSpeak(Dog{Name: "Rex"})      // "Woof!"
makeSpeak(Cat{Name: "Whiskers"}) // "Meow!"
makeSpeak(Human{Name: "Alice"})  // "Hello!"

// Useful built-in interfaces you'll see everywhere:
// error   — anything with Error() string
// io.Reader — anything with Read(p []byte) (n int, err error)
// io.Writer — anything with Write(p []byte) (n int, err error)
// fmt.Stringer — anything with String() string (like .toString())

Error Handling

In Go, errors are just values. If a function can fail, it returns an error. You check it. Every time. Yes, this is verbose — but it means bugs are obvious instead of hidden.

import (
    "errors"
    "fmt"
)

// Creating errors
err1 := errors.New("something went wrong")
err2 := fmt.Errorf("user %s not found", username)       // formatted
err3 := fmt.Errorf("loading config: %w", originalErr)   // wraps another error

// THE pattern — check immediately after every call that can fail
data, err := os.ReadFile("config.json")
if err != nil {
    return fmt.Errorf("reading config: %w", err)
}

// Custom error type — gives callers structured info about what went wrong
type NotFoundError struct {
    Resource string
    ID       int
}

func (e *NotFoundError) Error() string {
    return fmt.Sprintf("%s with id %d not found", e.Resource, e.ID)
}

// Unwrap and inspect errors
var nfe *NotFoundError
if errors.As(err, &nfe) {
    fmt.Println("missing:", nfe.Resource)
}

// Check for a specific sentinel error
if errors.Is(err, os.ErrNotExist) {
    fmt.Println("file doesn't exist")
}

Goroutines — Doing Things at the Same Time

A goroutine is a lightweight concurrent function. You can run thousands of them. They're much cheaper than OS threads.

// Start a goroutine with the "go" keyword — it runs concurrently
go fmt.Println("I'm running in the background!")

// Anonymous goroutine
go func() {
    fmt.Println("also in the background")
}()   // Note the () at the end — you must call the function

// WaitGroup — wait for multiple goroutines to finish before continuing
import "sync"

var wg sync.WaitGroup

for i := 0; i < 5; i++ {
    wg.Add(1)            // Tell the WaitGroup one more goroutine is starting
    go func(n int) {
        defer wg.Done()  // Signal when this goroutine finishes
        fmt.Println("worker", n)
    }(i)                  // Pass i as argument — very important, don't capture it directly
}

wg.Wait()   // Block here until all goroutines call Done()
fmt.Println("all done")

// Mutex — prevent two goroutines from writing to the same thing at once
var mu sync.Mutex
counter := 0

go func() {
    mu.Lock()
    counter++
    mu.Unlock()
}()

Channels — Goroutines Talking to Each Other

// Make a channel — like a typed pipe between goroutines
ch := make(chan string)

// Send into a channel (blocks until someone receives)
go func() {
    ch <- "hello from goroutine"
}()

// Receive from a channel (blocks until someone sends)
msg := <-ch
fmt.Println(msg)

// Buffered channel — holds up to N values before blocking
jobs := make(chan int, 5)
jobs <- 1
jobs <- 2
jobs <- 3

// Close a channel when done sending — receivers can detect this
close(jobs)

// Range over a channel until it's closed
for job := range jobs {
    fmt.Println("processing job", job)
}

// select — wait on multiple channels at once (like a switch for channels)
select {
case msg := <-ch1:
    fmt.Println("got from ch1:", msg)
case msg := <-ch2:
    fmt.Println("got from ch2:", msg)
case <-time.After(1 * time.Second):
    fmt.Println("timed out")
}

JSON

JSON is everywhere in web APIs. Go handles it through struct tags — special annotations that tell the JSON encoder what to call fields.

import "encoding/json"

// Struct with JSON tags
type Product struct {
    ID       int     `json:"id"`
    Name     string  `json:"name"`
    Price    float64 `json:"price"`
    InStock  bool    `json:"in_stock"`
    Internal string  `json:"-"`  // "-" means exclude from JSON
    Optional string  `json:"desc,omitempty"`  // omit if empty
}

// Encode Go → JSON
p := Product{ID: 1, Name: "Widget", Price: 9.99}
data, err := json.Marshal(p)
// data is []byte: {"id":1,"name":"Widget","price":9.99,"in_stock":false}

// Pretty-print JSON
data, err = json.MarshalIndent(p, "", "  ")

// Decode JSON → Go
jsonStr := `{"id":2,"name":"Gadget","price":19.99}`
var product Product
err = json.Unmarshal([]byte(jsonStr), &product)
// product.Name is now "Gadget"

// Decode JSON from an HTTP response body
var result Product
json.NewDecoder(resp.Body).Decode(&result)

⚠ DANGER ZONE — Emacs Commands That Delete Without Warning

Emacs predates the concept of a Trash/Recycle Bin. When you delete something in Dired, it goes straight to oblivion. No second chances.

Dired Deletion — There is No Trash Can

Key in DiredWhat it does — and why it's dangerous
dMark a file for deletion (just marks it — nothing deleted yet)
xExecute pending deletions — PERMANENTLY deletes everything marked with d. It asks once. After you say yes, it's gone.
DImmediately delete the file under cursor. No mark step. One key, it's gone. Use with extreme care.

Safe workflow: Use d to mark things, look carefully at what's marked, then press x. If you marked the wrong thing, press u to unmark it, or U to unmark everything and start over.

Overwriting Files

If you edit a file and press C-x C-s, it overwrites the file on disk immediately. Before saving, use C-/ to undo any changes you don't want. If you already saved and regret it, and you're using git, git checkout -- filename can restore it. Without git, it may be gone.
